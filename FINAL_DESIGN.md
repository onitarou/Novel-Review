# 小説共有・批評サイト 最終設計書

この文書は、`PROJECT.md`および`DESIGN_UNCLEAR_POINTS.md`で確定した内容を統合した最終設計書である。

# 1. 目的

本システムは、数十名から百名程度の限られたサークル内読者に対して、小説作品を限定公開し、本文に紐づく批評コメントを受け取るための個人向けWebサイトである。

大規模な投稿サイトやSNSではなく、管理者である投稿者が作品を管理し、共有URLを知っている読者だけが閲覧・コメントできることを重視する。

# 2. 利用者と権限

## 2.1 投稿者

投稿者は現時点では管理者1名のみとする。

主な権限:

- 管理者ログイン
- 作品の作成・編集・削除
- 章の作成・編集・削除
- 話の作成・編集・削除
- `.txt`ファイルからの本文取り込み
- あらすじと作者メモの編集
- 話ごとの公開状態管理
- 初回公開および再公開
- 作品共有URLの手動発行・手動停止・手動再発行
- コメント一覧の閲覧
- コメント状態の変更
- 版管理・改稿履歴の確認
- 作品本文とコメントのバックアップエクスポート

## 2.2 読者

読者はアカウントを持たない。管理者から共有された作品URLを使って作品を閲覧する。

主な権限:

- 共有URLが有効な作品の閲覧
- 公開状態が`限定公開`の話の閲覧
- コメント投稿
- 自分が投稿したコメントの編集・削除
- 縦書き・横書きなどの読書設定保存

読者識別は、ブラウザ単位の匿名IDと表示名で行う。表示名はコメント投稿時に入力し、ブラウザに保存する。

# 3. 認証・アクセス制御

## 3.1 管理者認証

初期実装では単一管理者ログインを採用する。

- 管理者パスワードは環境変数で設定する。
- 管理画面はサーバー側セッションで保護する。
- 将来的な複数管理者化を見越し、DBには`users`相当の構造を用意できる設計にする。

## 3.2 読者アクセス

読者は共有URLに含まれる推測困難なトークンで作品へアクセスする。

- 共有URLが有効な場合のみ作品を閲覧できる。
- 共有URLが停止済みの場合は作品内容を表示しない。
- 読者には`限定公開`の話だけを表示する。
- `下書き`および`非公開`の話は、読者には存在しないものとして扱う。

# 4. 作品構造

## 4.1 作品

作品は以下の情報を持つ。

- タイトル
- あらすじ
- 作者メモ
- 作成日時
- 更新日時
- 削除日時

あらすじは1000文字以内とする。作者メモは読者にも表示される公開メモとして扱う。

## 4.2 章

章は作品に属し、話を分類するために使う。

- 章タイトル
- 並び順
- 作成日時
- 更新日時
- 削除日時

章に属さない話も許可する。

## 4.3 話

話は作品に属し、任意で章に属する。

- 話タイトル
- 本文
- 公開状態
- 並び順
- 現在公開版番号
- 作成日時
- 更新日時
- 削除日時

公開状態は以下の3種類とする。

- `下書き`：投稿者のみ閲覧可能な編集途中状態。
- `限定公開`：有効な共有URLを持つ読者が閲覧できる状態。
- `非公開`：投稿者のみ閲覧可能で、完成済みでも読者には表示しない状態。

# 5. 本文記法

本文は記法付きプレーンテキストを正本として保存する。

## 5.1 ルビ

ルビは以下の形式で記述する。

```text
{振り仮名}{ふりがな}
```

前段を対象文字列、後段をふりがなとして扱う。

## 5.2 傍点

傍点は、後段が`・`または`﹅`だけの場合に表現する。

```text
{強調したい語}{・}
```

## 5.3 エスケープ

本文中に通常文字として波括弧を書きたい場合は、`\{`と`\}`でエスケープする。

## 5.4 レンダリング

保存時にはHTMLを保存しない。表示時に本文記法を安全なHTMLへ変換する。

# 6. 版管理・改稿履歴

話の版は公開済み本文を基準に記録する。

- 初回公開時の本文を`ver1`とする。
- 以降、編集後に管理者が再公開操作を行うたびにバージョンをインクリメントする。
- 下書き保存、自動保存、公開停止、共有URLの停止・再発行では本文バージョンを増やさない。
- コメントは投稿時点で公開されている版に紐づける。

各版は以下の情報を持つ。

- 話ID
- 版番号
- 本文
- 変更メモ
- 作成日時
- 公開日時

投稿者は過去版の本文全文と、版ごとのコメントを確認できる。差分表示は将来拡張とする。

# 7. 共有URL

共有URLは作品単位で管理する。

- 初期実装では、作品ごとに有効な共有URLは1本だけとする。
- 発行、停止、再発行はすべて管理者の手動操作とする。
- 自動期限切れは設けない。
- 再発行時は既存の有効URLを停止し、新しいURLを発行する。
- 停止済みURLでは作品を閲覧できない。

共有URLトークンは十分に長いランダム値とし、本番ではハッシュ化してDB保存する。

# 8. コメント

## 8.1 投稿

読者は`限定公開`の話に対してコメントを投稿できる。

コメントには以下を記録する。

- 対象作品
- 対象話
- 対象本文の版
- コメント本文
- 読者表示名
- 読者匿名ID
- 引用本文
- 引用開始位置
- 引用終了位置
- 引用前後の文脈
- コメント状態
- 作成日時
- 編集日時
- 削除日時

## 8.2 引用コメント

読者が本文の一部を選択した状態でコメントを投稿した場合、選択範囲を引用として保持する。

本文位置は、投稿時点の正規化済み本文に対する`start_offset`と`end_offset`で保存する。あわせて引用本文と前後文脈も保存し、改稿後の再探索に備える。

## 8.3 ハイライト

本文位置のハイライトは常時表示しない。

- コメントを選択したときだけ、対応する本文範囲をハイライトする。
- 選択解除時にはハイライトを消す。
- コメント一覧から話ページへジャンプした場合は、該当コメントを自動選択する。

## 8.4 編集・削除

読者は自分が投稿したコメントを編集・削除できる。

- 所有者判定には、ブラウザ単位の匿名IDとコメント投稿時に発行する編集用シークレットを使う。
- コメント編集時は`edited_at`を更新し、コメント状態を`未読`へ戻す。
- コメント削除時は物理削除せず、`deleted_at`を記録する。
- 削除済みコメントは通常画面とコメント一覧では非表示にする。
- 初期実装では、削除済みコメントの本文は空にし、削除日時と関連先だけを残す。

## 8.5 コメント状態

コメント状態は投稿者のみ変更できる。

- `未読`：投稿者がまだ確認していない。
- `確認済み`：投稿者が読んだが、改稿対応は未判断。
- `対応済み`：改稿または判断が完了している。
- `保留`：後で再検討する。

新規コメントの初期状態は`未読`とする。

## 8.6 コメント公開範囲

初期実装では、全コメントを閲覧できるのは投稿者のみとする。

読者はコメント作成と、自分のブラウザから投稿したコメントの確認・編集・削除のみ可能とする。他の読者のコメントは見えない。

# 9. 表示設計

## 9.1 管理者向け作品一覧

管理者のみ閲覧可能。

- 全作品を一覧表示する。
- 作品クリックで作品管理ページへ遷移する。
- コメント未読数、共有URL状態、更新日時を表示する。

## 9.2 作品ページ

投稿者と共有URLを持つ読者が閲覧できる。

- 左サイドバーに章と話を表示する。
- 章が設定されている場合は章の区分を表示する。
- 話未選択時は、メイン領域にあらすじと作者メモを表示する。
- 読者には`限定公開`の話だけを表示する。

## 9.3 話ページ

本文を表示する。

- 左サイドバーは維持する。
- 縦書き・横書きの両方に対応する。
- 横書き表示時は右サイドバーにコメント領域を置く。
- 縦書き表示時は本文下部にコメント領域を置く。
- 画面幅が狭い場合は、横書きでもコメント領域を本文下部に移動する。

## 9.4 読書設定

読書設定は読者のブラウザ単位で保存する。

- 縦書き・横書き
- 文字サイズ
- 行間
- テーマ

初期実装ではlocalStorageを使い、DBには保存しない。

## 9.5 コメント一覧ページ

管理者のみ閲覧可能。

- 作品横断でコメントを一覧できる。
- 作品、章、話、読者名、作成日時、コメント状態で絞り込み・並び替えできる。
- コメント状態を変更できる。
- コメントから対象話と引用位置へジャンプできる。

# 10. バックアップエクスポート

管理者は作品本文とコメントをバックアップとしてエクスポートできる。

初期実装ではZIPまたはJSONを想定し、以下を含める。

- 作品メタデータ
- 章
- 話
- 現在版本文
- 過去版本文
- コメント
- コメント状態
- 引用情報
- 関連する本文版

文字コードはUTF-8に統一する。

# 11. DB設計

本番DBはTursoを使用する。TursoはSQLite互換であるため、初期実装ではSQLite互換SQLでスキーマを定義する。

主要テーブル:

- `users`
- `works`
- `chapters`
- `stories`
- `story_versions`
- `share_links`
- `comments`
- `reader_identities`

読書設定は初期実装ではDB保存せず、ブラウザのlocalStorageに保存する。

全テーブルの日時はUTCで保存し、画面表示では日本時間に変換する。

# 12. ルーティング

## 12.1 管理者

- `GET /admin/login`
- `POST /admin/login`
- `POST /admin/logout`
- `GET /admin/works`
- `GET /admin/works/new`
- `POST /admin/works`
- `GET /admin/works/:workId`
- `POST /admin/works/:workId`
- `POST /admin/works/:workId/share/issue`
- `POST /admin/works/:workId/share/revoke`
- `POST /admin/works/:workId/share/regenerate`
- `GET /admin/stories/:storyId`
- `POST /admin/stories/:storyId/draft`
- `POST /admin/stories/:storyId/publish`
- `GET /admin/comments`
- `POST /admin/comments/:commentId/status`
- `GET /admin/export`

## 12.2 読者

- `GET /s/:token`
- `GET /s/:token/stories/:storyId`
- `POST /s/:token/stories/:storyId/comments`
- `POST /s/:token/comments/:commentId/edit`
- `POST /s/:token/comments/:commentId/delete`

# 13. 削除方針

作品、章、話、コメント、版履歴は原則として論理削除する。

- 通常画面では`deleted_at`が入ったデータを非表示にする。
- コメントや版履歴との整合性を守る。
- 物理削除は将来の管理者向けメンテナンス操作とする。

# 14. セキュリティ

- 管理画面の変更系操作にはCSRF対策を行う。
- 読者コメント投稿にもページ表示時のトークンを利用する。
- 共有URLトークンは推測困難な値にする。
- 下書き・非公開話への読者アクセスは404相当とする。
- コメント本文と本文記法のHTML変換時には必ずエスケープする。

# 15. 初期実装範囲

最初の実装では、以下を優先する。

1. 管理者ログイン
2. 作品一覧
3. 作品詳細
4. 話の編集
5. 初回公開・再公開による版管理
6. 共有URLの発行・停止・再発行
7. 読者向け作品ページ・話ページ
8. 本文記法レンダリング
9. コメント投稿
10. 読者自身のコメント編集・削除
11. コメント選択時の本文ハイライト
12. 管理者向けコメント一覧と状態変更
13. JSON形式のバックアップエクスポート

この初期実装では、外部パッケージに依存しないNode.jsアプリとして開始し、DBはSQLite互換SQLで実装する。Turso接続は、本番環境構築時にDBアダプタを差し替える。
